# postgresql컬럼순서 변경안되는 단점때문에 mysql/mariaDB를 AWS로 사용한다고 하면 다음 점 고려해야함

1. 통합의 편의성
   Postgres(Neon, Vercel DB) 쪽이 Vercel과 통합이 더 쉽습니다.
   MySQL/MariaDB는 직접 드라이버 설치하고 연결 설정을 더 신경써야 합니다.
2. 코드 변경
   현재 glucose.js는 PostgreSQL SQL 문법을 기준으로 작성됐습니다.
   MySQL/MariaDB로 바꾸면 SQL 문법, 연결 라이브러리(mysql2, sequelize, Prisma 등)를 바꿔야 합니다.
3. 보안 및 네트워크
   AWS RDS/MariaDB를 쓰면 VPC, 보안 그룹, 퍼블릭 IP/프라이빗 엔드포인트, SSL 등을 직접 설정해야 합니다.
   Vercel과 외부 DB 간 네트워크 지연도 고려해야 합니다.

# 앱개발 방식

1. 네이티브 개발: Android는 Kotlin/Java, iOS는 Swift/Object-C
   성능은 가장 좋고, 플랫폼 API 접근도 자유롭지만 두 플랫폼은 각각 따로 개발해야 하니 비용과 리소스가 많이 듭니다.
   그렇다면 여기서 말하는 성능은 어떤게 있을가요?
   네이티브 코드는 OS 레벨에서 바로 실행되기 때문에, CPU, GPU, 메모리, 센서 등 하드웨어 리소스에 직접 접근이 가능합니다. 즉 카메라 제어, 블루투스, 저전력 최적화 등 모바일에 대한 제어권을 완전히 가지고 있죠. 반면에 네이티브로 작성되지 않은 코드들은 이런 API를 간접적으로 호출하거나, 플러그인으로 접근해야 하기 때문에 느리거나 제약이 생기는 경우가 많습니다.
   또한 렌더링과 멀티스레팅/백그라운드 제어에도 차이가 납니다. 네이티브는 GPU 가속을 직접 활용해서 UI 애니메이션, 전환, 3D 렌더링 같은 그래픽 연산을 바로 처리할 수 있지만 네이티브가 아닌 경우에는 브릿지, 네이티브 UI로 전달되기 때문에 체감 차이가 발생합니다. 또한 OS의 스레드와 프로세스 모델에 접근이 가능하기 때문에 비동기나 병렬 연산에서의 최적화가 가능하죠.

2. 하이브리드 앱: React Native, Flutter, Ionic, Capacitor 같은 프레임워크를 이용해서 하나의 코드로 Android와 iOS를 모두 빌드하는 방식입니다. 네이티브 코드와 웹 기술(HTML, JS, CSS)이 섞여 있어 브라우저 위에서 네이티브 껍데기를 입힌 방식으로 동작합니다. 기본적으로 웹뷰(Web View)안에 페이지를 띄우고, 이 웹뷰와 네이티브 코드가 브릿지(Bridge)를 통해 통신하는 구조

3. 웹앱
   웹 기반이라 배포나 업데이트가 진짜 간단, 오프라인 상태에서도 일부 페이지 접근 가능, 푸쉬 알림, 앱처럼 아이콘/스플래시 화면 표시가 되

실제로 쓰는 앱은 대부분 1,2,3이나 1,3을 혼합해서 씀

# react vs reactNative

공유하는 부분은 2-30% 밖에 안되고, 근본적으로 다른 아키텍쳐
React (웹)
└── HTML 렌더링
├── <div> → DOM
├── CSS 스타일링
└── 브라우저 API

React Native (앱)
└── 네이티브 렌더링
├── <View> → iOS/Android UI
├── StyleSheet 스타일링
└── 네이티브 API

React Native/혹은 native가 꼭 필요한 경우

- 성능이 중요한 경우 예)게임,영상편집,실시간채팅앱
- 오프라인 기능 중요 예) 지도, 금융, 항공사
- 배터리 성능 중요한 경우
- 보안: 암호화 모듈(하드웨어 보안), 지문/얼굴인식같은 생체인증, 탈옥/루팅 감지,
  보안 엔클레이브(Secure Enclave), 앱 무결성 검증?

보안 엔클레이브(Secure Enclave): 스마트폰 내부의 "금고", 특수한 보안 칩 → 민감한 정보만 저장
스마트폰
├── 일반 프로세서 (CPU)
│ └── 일반 앱들이 실행
│ (우리 코드, 해킹 위험 있음)
│
└── 보안 엔클레이브 (별도 칩) ← 격리됨
└── 지문, 얼굴, 비밀번호 저장
(해킹 불가능)

Gmail 네이티브 앱이 있는 이유

1. 성능
   └─ 웹앱: 메일 로드 2-3초
   └─ 네이티브: 메일 로드 0.5초

2. 배터리 효율
   └─ 새 메일 알림 24/7 대기
   └─ 네이티브만 최적화 가능

3. 오프라인 기능
   └─ 이전 메일 로컬 저장
   └─ 인터넷 끊겨도 메일 읽기 가능

4. 푸시 알림
   └─ OS 레벨 알림 (매우 중요)
   └─ 웹앱보다 안정적

pwa로 지원되는 것

- 앱설치하여 바탕화면에 저장 - 주요 이유, 나머지는 지원이 제한적이거나 있어도 성능떨어지거나 배터리소모가 많음
- push알림 - 배터리소모 많음
- 오프라인 화면
- 앱스토어 올리기
  오프라인 실행: Service Worker + 캐싱(IndexedDB)로 오프라인 동작
  Background Sync: 네트워크 복구 시 백그라운드 동기 동기화(제한적 지원)
  Periodic Sync: 주기적 백그라운드 작업(브라우저 지원 제한적)
  Badging API: 홈화면 아이콘에 배지 표시(지원 브라우저 제한)
  Web Share / Share Target: 다른 앱에서 내 PWA로 공유 처리
  Media Session API: 미디어 제어(잠금화면, 알림 컨트롤)
  Screen Wake Lock: 화면 꺼짐 방지(장기간 측정 등)
  File System Access: 로컬 파일 읽기/쓰기(브라우저 지원 제한)
